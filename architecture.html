Technical overview

Vault architecture

Vault is an offline password manager that runs entirely inside the browser. Credentials are encrypted locally using the browser’s Web Crypto API and stored on the user’s device. No passwords are transmitted to a server and no cloud account infrastructure is required.

Because Vault is a security-related application, the full source code is publicly available so users can inspect the implementation and verify the trust model.

Public source code

Transparency is important for security tools. The Vault source code is available on GitHub so users and developers can review how local encryption, browser storage, export, import, and offline behaviour are implemented.

View the Vault repository on GitHub

Core model

  • Client-side encryption using the Web Crypto API.
  • Local persistence using browser storage on the device.
  • Manual export and import of encrypted vault files.
  • Offline capability through PWA installation and a service worker.

Design intent

The architecture is intentionally small and understandable. Vault avoids server infrastructure, account systems, or hosted password databases. Instead it focuses on a simple local-first model where encryption and storage occur on the user’s device.

Application shape

  • app.html – application shell and runtime entry point.
  • app.js – main application controller.
  • storage.js – local vault storage, export and import.
  • crypto.js – encryption helpers built on Web Crypto.
  • UI modules – entry rendering, dialogs, icons, search, reset, and auto-lock behaviour.
  • service worker – enables PWA offline behaviour.

Security boundaries

Vault reduces dependence on third-party infrastructure, but it does not claim to protect against compromised devices, weak master passwords, unsafe backups, or browser vulnerabilities.

Practical summary

Vault follows a local-first security model: encryption occurs in the browser, storage remains on the device, and backups are moved deliberately by the user. The codebase is intentionally small so the behaviour of the application can be inspected and understood.
